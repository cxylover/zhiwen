# 计数缓存优化方案

日期：2026-06-08

## 背景

原点赞流程中，点赞或取消点赞后会通过 Spring 本地事件同步更新被点赞知文所在的 feed 页面缓存：先通过反向索引找到页面 key，再反序列化整页 JSON，修改目标知文的 `likeCount` 或 `favoriteCount`，最后重新序列化写回 Redis 和本地 Caffeine。

这种做法的问题是，点赞/收藏属于高频动态数据，一旦热点知文出现在多个页面或多个实例缓存中，每次点赞都会触发大量缓存读写，容易形成缓存更新风暴。

## 核心思路

新方案将知文展示数据拆成两类：

- 静态数据：标题、描述、封面、标签、作者信息、正文 URL、可见性、置顶状态等。
- 动态数据：点赞数、收藏数、当前用户是否点赞、当前用户是否收藏。

feed 流缓存和详情页缓存只保存静态数据；动态数据不落入页面缓存，而是在接口返回前运行时注入。

动态数据来源：

- `liked/faved`：从 Redis bitmap 事实层实时读取。
- `likeCount/favoriteCount`：从 Redis SDS 计数快照读取。
- Kafka 聚合桶：承接点赞/收藏增量，异步 flush 到 SDS。

## 点赞全流程

1. 用户调用点赞接口。

   ```text
   POST /api/v1/actions/like
   ```

2. `CounterService.like(...)` 执行 Redis Lua 脚本。

   Lua 基于用户 ID 定位 bitmap 分片，并通过 `GETBIT/SETBIT` 原子判断状态是否变化。

   - 如果用户已经点过赞，返回 `changed=false`，不会重复产生计数事件。
   - 如果用户之前没有点赞，bitmap 事实层立即更新，返回 `changed=true`。

3. 状态变化后发布事件。

   - Kafka `CounterEvent`：用于异步聚合知文维度计数。
   - Spring 本地 `CounterEvent`：用于本 JVM 内快速执行旁路逻辑。

4. Kafka 消费端聚合计数。

   Kafka 消费端将增量先写入聚合桶，再批量 flush 到 SDS：

   ```text
   agg:{schema}:knowpost:{postId}
   cnt:{schema}:knowpost:{postId}
   ```

5. Spring 本地监听器处理旁路逻辑。

   - 根据 `postId` 查询作者。
   - 更新作者收到的点赞数或收藏数。
   - 通过反向索引找到受影响的本地 feed 页缓存并失效。

   点赞/收藏事件不再反序列化 feed 页面，也不再写回页面 JSON。

6. 下次查询 feed 或详情页时，返回前运行时注入动态字段。

   - 批量读取 SDS 注入 `likeCount/favoriteCount`。
   - 查询 bitmap 注入 `liked/faved`。

## Feed 缓存分层

### L1：Caffeine 本地页缓存

缓存 key 示例：

```text
feed:public:{size}:{page}:v1
feed:mine:{userId}:{size}:{page}
```

缓存 value：

```text
FeedPageResponse
```

每个 `FeedItemResponse` 只缓存静态字段：

```text
id
title
description
coverImage
tags
authorAvatar
authorNickname
tagJson
isTop
```

不缓存：

```text
likeCount
favoriteCount
liked
faved
```

### L2：Redis ID 页缓存

缓存 key 示例：

```text
feed:public:ids:{size}:{hourSlot}:{page}
feed:public:ids:{size}:{hourSlot}:{page}:hasMore
```

L2 只保存页面骨架：

- 当前页知文 ID 列表。
- `hasMore` 标志。

它不保存标题、作者、封面、计数或用户态。

`hourSlot` 用于按小时切片，避免同一批页面缓存集中失效。

### L3：Redis item 片段缓存

缓存 key 示例：

```text
feed:item:{postId}
```

缓存单条知文在 feed 中展示所需的静态片段：

```text
id
title
description
coverImage
tags
authorAvatar
authorNickname
tagJson
isTop
```

不保存：

```text
likeCount
favoriteCount
liked
faved
```

### 反向索引

反向索引不属于 L1-L3，但用于定位页面：

```text
feed:public:index:{postId}:{hourSlot}
```

它记录某个知文出现在哪些 feed 页中。静态字段变化时，可以通过它快速定位受影响页面并失效缓存。

在新方案中，点赞/收藏变化不依赖反向索引更新页面计数。点赞/收藏数据由返回前运行时注入。

## 详情页缓存

详情页缓存也保留两级缓存：

### L1：Caffeine 本地详情缓存

```text
key = knowpost:detail:{id}:v1
value = KnowPostDetailResponse 静态版
```

### Redis：跨实例详情缓存

```text
key = knowpost:detail:{id}:v1
value = KnowPostDetailResponse 静态版 JSON
```

详情页缓存保存：

```text
id
title
description
contentUrl
images
tags
authorId
authorAvatar
authorNickname
authorTagJson
isTop
visible
type
publishTime
```

详情页缓存不保存：

```text
likeCount
favoriteCount
liked
faved
```

查询详情页时：

1. 先查 Caffeine。
2. Caffeine 未命中则查 Redis。
3. Redis 命中后反序列化静态详情，并回填 Caffeine。
4. Redis 未命中则回源 DB，构造静态详情，同时写 Redis 和 Caffeine。
5. 返回前实时注入点赞数、收藏数、当前用户点赞态和收藏态。

## Feed 缓存重建过程

当查询公共 feed 时：

1. 先查 L1 Caffeine。

   命中后拿到静态页，返回前执行动态数据注入。

2. L1 未命中时，尝试用 L2 + L3 组装页面。

   - 从 L2 读取当前页 ID 列表。
   - 批量读取 L3 中的 `feed:item:{postId}`。
   - 如果所有 item 片段齐全，则组装静态页，回填 L1，并在返回前注入动态数据。

3. L2 或 L3 缺失时回源 DB。

   使用 single-flight 锁保证同一页同一时刻只有一个线程回源。

   回源后写入：

   - Redis ID 页缓存。
   - Redis `hasMore` 缓存。
   - Redis item 静态片段缓存。
   - Redis 反向索引。
   - Caffeine 静态页缓存。

4. 返回前注入动态数据。

   - `getCountsBatch("knowpost", ids, List.of("like", "fav"))` 批量读取 SDS。
   - `isLiked/isFaved` 查询当前用户 bitmap 状态。

## 详情页缓存重建过程

当查询详情页时：

1. 先查本地 Caffeine。
2. 本地未命中则查 Redis。
3. Redis 命中后回填 Caffeine。
4. Redis 未命中则回源 DB。
5. DB 回源后写入 Redis 和 Caffeine。
6. 返回前统一注入动态字段。

详情页缓存失效由静态字段变化触发，例如：

- 内容上传确认。
- 标题、描述、标签、图片等元数据变化。
- 可见性变化。
- 置顶状态变化。
- 删除。

点赞/收藏不触发详情页缓存失效。

## 一致性分析

### 用户态一致性

`liked/faved` 来自 Redis bitmap。点赞接口同步更新 bitmap，因此用户点赞后再次查询，用户态可以立即准确反映。

### 计数一致性

`likeCount/favoriteCount` 来自 SDS，而 SDS 由 Kafka 聚合异步更新，因此计数是最终一致。

这意味着：

- `liked=true` 可以立即出现。
- `likeCount` 可能在 Kafka 聚合 flush 前短暂滞后。

如果产品要求点赞后计数强实时，可以考虑：

- 查询时读取 SDS 并叠加聚合桶中的未 flush 增量。
- 或在点赞同步路径直接更新 SDS，再让 Kafka 做校正或异步落后处理。

## 前端乐观更新方案

在当前后端架构下，点赞数是最终一致的，`liked/faved` 用户态来自 bitmap，可同步准确更新。为了让用户点击点赞后立即看到反馈，前端可以采用乐观更新。

这里的“乐观更新”不是乐观锁。前端不负责并发锁定，只是在本地先更新 UI；真正的幂等控制仍然由后端 Redis Lua + bitmap 保证。

### 基本流程

1. 用户进入知文 A 详情页。

   前端请求详情页，得到初始状态：

   ```text
   likeCount = 100
   liked = false
   ```

2. 用户点击点赞按钮。

3. 前端保存点击前快照。

   ```text
   beforeLikeCount = 100
   beforeLiked = false
   ```

4. 前端立即更新 UI。

   ```text
   likeCount = 101
   liked = true
   ```

5. 前端向后端发起点赞请求。

   ```text
   POST /api/v1/actions/like
   ```

6. 后端执行 bitmap 幂等切换，并返回结果。

   响应示例：

   ```json
   {
     "changed": true,
     "liked": true
   }
   ```

7. 前端根据后端结果决定是否保持、修正或回滚 UI。

### 成功处理

如果后端返回：

```json
{
  "changed": true,
  "liked": true
}
```

说明本次点赞确实生效，前端保持乐观更新后的 UI。

```text
likeCount = beforeLikeCount + 1
liked = true
```

如果后端返回：

```json
{
  "changed": false,
  "liked": true
}
```

说明后端认为用户此前已经点过赞，本次请求没有产生新的状态变化。此时前端不应该继续叠加计数，可以采用以下策略之一：

- 将 `liked` 修正为后端返回值。
- 将 `likeCount` 回滚到点击前值，或触发一次轻量计数刷新。
- 如果前端确认当前 UI 已经是 `liked=true`，则不再重复 +1。

推荐以“点击前快照 + 后端状态”为准，避免重复点击导致本地计数多加。

### 失败回滚

如果点赞请求失败，例如网络错误、鉴权失败、服务端异常，前端应回滚到点击前快照：

```text
likeCount = beforeLikeCount
liked = beforeLiked
```

如果失败原因不明确，也可以在回滚后重新请求详情页或计数接口进行校准。

### 取消点赞

取消点赞同理。

初始状态：

```text
likeCount = 101
liked = true
```

用户点击取消点赞后，前端立即更新：

```text
likeCount = 100
liked = false
```

后端成功且 `changed=true` 时保持 UI；失败时回滚。

计数变化需要做下限保护：

```text
likeCount = max(0, beforeLikeCount - 1)
```

### 连续点击与乱序响应

前端需要处理用户快速连续点击的问题，例如：

```text
点赞 -> 取消点赞 -> 点赞
```

如果多个请求同时在路上，旧响应可能晚于新响应返回，从而覆盖最新 UI。

可选策略：

1. 请求 pending 时禁用按钮。

   这是最简单稳定的方案。一次点赞或取消点赞完成后，才允许下一次点击。

2. 使用本地操作版本号。

   每次点击生成一个递增版本号或 requestId。响应返回时，只有与当前最新版本一致的响应才允许更新 UI。

3. 使用目标状态接口。

   前端不发送“切换”，而是发送目标状态：

   ```text
   set liked = true
   set liked = false
   ```

   后端根据目标状态执行点赞或取消点赞。这样更适合处理连续点击和重试。

当前后端已经提供 `like/unlike` 两类接口，前端可以根据本地目标状态选择调用对应接口。

### 与后端最终一致的关系

乐观更新只影响当前用户当前页面的局部 UI，不改变后端事实层。

最终事实仍由后端保证：

- bitmap 保证用户是否点赞的事实正确。
- Kafka 聚合最终写入 SDS，保证总计数最终一致。
- 下次刷新详情页、feed 页或重新获取计数时，前端会拿到后端最终计数。

因此可能出现短暂状态：

```text
当前用户页面显示 likeCount = 101
其他用户页面仍显示 likeCount = 100
```

这是可接受的局部不一致，换取用户点击后的即时反馈。

### 推荐前端状态机

前端可以为每个知文维护以下状态：

```text
liked
likeCount
pending
lastRequestId
snapshot
```

点赞时：

```text
snapshot = { liked, likeCount }
pending = true
liked = true
likeCount = likeCount + 1
requestId = nextRequestId()
send like request
```

成功时：

```text
if requestId is latest:
    pending = false
    liked = response.liked
    if response.changed == false:
        optionally refresh count
```

失败时：

```text
if requestId is latest:
    pending = false
    liked = snapshot.liked
    likeCount = snapshot.likeCount
```

取消点赞时流程相同，只是目标状态改为 `liked=false`，计数本地先减 1。

### 接口返回建议

当前点赞接口返回 `changed` 和 `liked` 已经可以支持乐观更新。

如果后续希望前端更容易校准，可以考虑让接口额外返回后端当前状态：

```json
{
  "changed": true,
  "liked": true,
  "serverTime": "2026-06-08T21:30:00Z"
}
```

如果要返回 `likeCount`，需要明确它的语义：

- 返回 SDS 当前值：可能滞后，但与后端查询一致。
- 返回 SDS + 聚合桶增量：更接近实时，但实现复杂度更高。
- 返回本次操作后的用户视角估算值：体验好，但不应作为全局强一致计数。

在当前方案中，推荐点赞接口先不强求返回最新总计数，由前端乐观更新和后续刷新校准即可。

## 方案收益

1. 避免缓存更新风暴。

   点赞/收藏不再触发整页 JSON 反序列化、修改、序列化和写回。

2. 缓存职责更清晰。

   页面缓存只负责静态展示数据，动态数据由计数系统和 bitmap 系统负责。

3. 热点内容更稳定。

   高频点赞不会导致热点 feed 页反复失效或重写。

4. 支持多用户共享缓存。

   `liked/faved` 不落入公共缓存，避免用户态污染。

5. 更适合多实例部署。

   Redis 作为共享静态缓存，Caffeine 作为本实例热点副本；动态数据统一从 Redis 事实层和计数层读取。

## 后续优化建议

1. 将 `isLiked/isFaved` 做批量 pipeline。

   目前计数读取已经有 `getCountsBatch`，但用户态仍是逐条 bitmap 查询。可以新增批量接口减少 RTT。

2. 回源写 Redis 使用 pipeline。

   feed 回源写入 ID 页、item 片段、反向索引和 TTL 时，可以先完成 JSON 序列化，再通过 pipeline 批量写入 Redis。

3. 静态字段变化时同时失效 feed item 片段。

   当标题、封面、作者信息等 feed 展示字段变化时，需要确保 `feed:item:{postId}` 失效或重建。

4. 检查详情页权限缓存边界。

   详情缓存 key 当前只按 `postId` 维度构造。如果非公开内容也进入共享缓存，需要确保缓存命中路径仍然执行权限校验，避免权限绕过。
